## What this service actually does

MinIO acts as the object storage backend for the Milvus vector database. It stores the actual vector embeddings, scalar payloads, and index files generated by the RAG pipeline.

The fleet uses it exclusively via `milvus-standalone` over the internal `db-vector` network. `milvus-standalone` connects via `MINIO_ADDRESS` env var.

It is NOT used for general file storage (e.g., user uploads, logs). Those reside on ZFS/NFS elsewhere. Do not route general traffic here.

### Configuration shape

Standard MinIO server mode. `command: ["server", "/minio_data", "--console-address", ":9001"]`.
API exposed on port 9000 (internal only). Console on 9001 (exposed via `db-vector` network, not host ports).
Credentials driven by `MINIO_ROOT_USER` and `MINIO_ROOT_PASSWORD` env vars.

- `openwebui` — Indirectly. RAG retrieval fails if Milvus cannot read from MinIO.
- `vane` — Indirectly. AI search fails.

## State & persistence

- **Volume Name:** `ai_minio_data` (managed by Docker volume driver).
- **Mount Path:** `/minio_data` inside container.
- **Data Content:** Milvus object store files (buckets `by-dev`, etc.).
- **Growth Rate:** Linear with document ingestion. 1GB ≈ ~10k-50k chunks depending on embedding dim.
- **Backup:** No explicit backup job defined in compose. Relies on host-level volume snapshots (ZFS/Docker).
- **RPO/RTO:** ? (No automated backup policy recorded).
- **Replication:** None. Single volume mount.

## Failure modes

- **Symptom:** `milvus-standalone` healthcheck fails with 503.
- **Root cause:** MinIO disk full or permission denied on volume.
- **Mitigation:** `docker exec -it milvus-minio df -h`. Clear old buckets or extend volume.
- **Prevention:** Host-level disk monitoring on `tmtdockp01`.

- **Symptom:** Milvus connection refused.
- **Root cause:** MinIO container stopped or network bridge `db-vector` missing.
- **Mitigation:** `docker compose restart milvus-minio`. Check `docker network inspect db-vector`.
- **Prevention:** `depends_on` on `milvus-standalone` ensures startup order (but not runtime readiness).

- **Symptom:** Data corruption after host reboot.
- **Root cause:** Dirty shutdown (kernel panic/power loss) on volume.
- **Mitigation:** Restore from snapshot.
- **Prevention:** Enable `--sync` flags (not currently configured).

## HA constraints (input to fleet-reliability design)

- **Replicable?** No. Single-instance deployment in `milvus-standalone` architecture.
- **Coordination requirements:** None. MinIO runs in standalone mode, not distributed cluster mode.
- **State sharing across replicas:** N/A (Single instance).
- **Hardware bindings:** None (CPU-only). Runs on `tmtdockp01`.
- **Network constraints:** Port 9000 (API) and 9001 (Console) must be unique per host if multiple stacks exist. Currently bound to `db-vector` bridge only.
- **Cluster size constraints:** N/A.
- **Migration cost:** High. Moving to distributed MinIO or multi-replica Milvus requires data migration and schema changes in `milvus-standalone`.
- **SPOF:** Yes. This service is a single point of failure for the entire Vector DB stack.
- **Upgrade Impact:** Rolling restart stops writes. Milvus may buffer or error depending on client retry logic.

### Known sharp edges
- **Console Access:** Port 9001 is not published to host (`ports` block empty for MinIO). Must access via `docker exec` or internal network tunnel.
- **Env Var Defaults:** `MINIO_ACCESS_KEY` defaults to `minioadmin`. If not overridden in `.env`, credentials are public knowledge in the fleet.
- **Volume Backups:** No automated snapshot policy for `ai_minio_data`. Data loss on host disk failure is total.
