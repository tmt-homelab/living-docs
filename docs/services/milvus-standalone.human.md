## What this service actually does

Milvus standalone serves as the vector database backend for the AI Stack's Retrieval-Augmented Generation (RAG) pipeline. It stores and indexes embeddings generated by the `vllm-embed` service. OpenWebUI queries this service via HTTP/gRPC on port `19530` to retrieve relevant document chunks before passing them to the LLM (`vllm-chat`).

It is NOT used for primary application data, user authentication, or model weights storage. It relies on `milvus-minio` for object storage of index files and `milvus-etcd` for metadata coordination. It runs strictly on CPU (`NVIDIA_VISIBLE_DEVICES=` explicitly empty), despite residing on a GPU-heavy host (`tmtdockp01`).

## Dependencies

### Hard (service cannot start or operate without these)
- `milvus-etcd` — Stores metadata and cluster coordination info. Milvus fails to start if unreachable.
- `milvus-minio` — Stores vector index files and raw data blobs. Milvus fails read/write if unreachable.

### Soft (degrade if these are down, but service still functional)
- None (Direct dependencies are hard).

### Reverse (services that depend on THIS service)
- `openwebui` — Uses `MILVUS_URI=http://milvus-standalone:19530` for RAG. RAG retrieval fails silently or errors if Milvus is down.
- `litellm` — Indirectly via OpenWebUI; no direct connection in compose.

## State & persistence

State is distributed across three containers in this stack, creating a complex recovery surface.
- **Database tables / schemas:** Managed internally by Milvus. No direct SQL access.
- **Filesystem state:** `/var/lib/milvus` (Volume `ai_milvus_data`). Contains logs, WAL, and local cache. Growth rate depends on ingestion volume (currently unknown).
- **In-memory state:** All in-memory indexes are lost on restart. Re-indexing may occur depending on persistence settings.
- **External state:** `milvus-minio` volume `ai_minio_data` holds the actual vector data. `milvus-etcd` volume `ai_etcd_data` holds metadata. **Backup requires snapshotting all three volumes.**

## Failure modes

- **Dependency Mismatch:** Upgrading `milvus-standalone` without matching `milvus-etcd`/`minio` versions can corrupt metadata.
- **Disk Full:** `ai_milvus_data` or `ai_minio_data` fills up during heavy RAG ingestion. Service halts write operations.
- **Etcd Quorum Loss:** Single-instance Etcd has no quorum, but corruption requires manual etcd data reset.
- **MinIO Corruption:** Object store corruption renders vector data unreadable even if Etcd is healthy.

## HA constraints (input to fleet-reliability design)

- **Replicable?** No. Configured as `standalone`. Cannot scale horizontally without switching to `cluster` mode (requires re-architecting dependencies).
- **Coordination requirements:** Requires external Etcd (here: single container). Does not support leader election internally in standalone mode.
- **State sharing across replicas:** N/A (Single instance). If migrating to cluster, requires shared MinIO and Etcd cluster.
- **Hardware bindings:** CPU-only. Explicitly configured to ignore host GPUs (`NVIDIA_VISIBLE_DEVICES=`). Can be moved to non-GPU host without config change.
- **Network constraints:** Port `19530` (gRPC/HTTP), `9091` (metrics). No multicast/broadcast required.
- **Cluster size constraints:** Current setup is 1:1:1 (Milvus:Etcd:Minio). This is a triple single-point-of-failure (SPOF).
- **Migration cost:** High. Moving to clustered Milvus requires re-indexing all vectors or complex data migration scripts. Data in MinIO must be preserved and accessible by new Milvus nodes.
